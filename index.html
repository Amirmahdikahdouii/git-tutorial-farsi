آموزش گیت و گیتهاب

گیت چیه؟

  • گیت ابزاریه که مدریت پروژه های ما رو به عهده میگیره و امکانات زیر رو هم به ما میده:
  • - گیت امکان نگهداری و ذخیره کد های ما به صورت مرحله به مرحله رو داره
  • - گیت به افراد یه پروژه این قابلیت رو میده که بتونن با هم روی یک پروژه کار کنن
  • - گیت مدیریت کلی کد های ما رو انجام میده

چطوری بفهمیم گیت روی سیستم نصب هست یا نه؟

git --version
git version 2.34.1

نکته:

ما دستورات گیت رو توی محیط ترمینال سیستم عامل وارد میکنیم. اگه با ترمینال زیاد آشنا نیستید نگران نباشین، دستورات رو با هم بررسی میکنیم و دونه دونه میریم جلو

Terminal Inputes:

دستورات ترمینال به ورودی و خروجی های استاندارد یا stdin: Standard Input و stdout: Standard Output تقسیم میشن. ما دستورات ورودی رو از روی علامت شناسایی میکنیم. دستوراتی که اولشون این علامت هست رو میتونین کپی کنین و توی ترمینال سیستم تون وارد کنید و تست شون کنین. مثلا توی مثال بالا git --version یه دستور ورودیه و git version 2.34.1 یک خروجی استاندارد برای این دستور هست. اگه با هر پیامی غیر از این پیام رو به رو شدین احتمالا گیت روی سیتم تون نصب نیست و باید اون رو نصب کنین.

نصب گیت روی سیستم:

برای نصب کردن گیت روی سیستم، باید با توجه به اینکه از چه سیتم عاملی استفاده میکنیم، یکی از راه های نصب رو انتخاب کنیم. در ادامه ما نصب گیت روی سه سیستم عامل Windows, Mac, Linux رو توضیح میدیم که بسته به سیستم عاملی که دارین، از نحوه نصب گیت استفاده کنین.

Windows

برای نصب گیت روی ویندوز، از این لینک وارد بخش دانلود گیت برای ویندوز میشیم و برای ویندوز یا هر سیستم عامل دیگه ای میتونیم دانلود و نصب کنیم. توی این مرحله از شما انتظار میره که بتونین نرم افزار گیت رو روی سیستم تون نصب کنید.

نکته:

در ویندوز برای استفاده از گیت، ما از ترمینال مخصوصی که برای استفاده گیت روی ویندوز طراحی شده هم میتونیم استفاده کنیم.

Mac

برای نصب گیت روی مک هم مثل بالا میتونیم از این لینک برای نصب گیت اقدام کنیم.

Linux

برای نصب گیت روی توزیع اوبونتو که خیلی محبوب هست، ما یه ترمینال باز میکنیم و داخل ترمینال دستور sudo apt-get install git رو وارد میکنیم و گیت روی سیستم ما به راحتی نصب میشه.

نکته:

برای اینکه از نصب گیت روی سیستم مطمئن بشین، ترمینال رو باز کنید و از دستوری که بالاتر برای بررسی نسخه گیت روی سیستم استفاده شد استفاده کنید. اگه به خطا نرسیدید و بهتون ورژن گیت رو برگردوند تبریک میگم، شما گیت رو نصب کردین!

کانفیگ اولیه

برای اینکه روی سیستم به گیت خودمون رو معرفی کنیم، تا از این اطلاعات ما توی هر کامیت استفاده بشه، باید به صورت زیر عمل کنیم.

برای معرفی Username

git   config   --global   user.name   "your username"

برای معرفی Email

git   config   --global   user.email   "yourEmail@example.com"

نکته:

از آپشن --global برای معرفی این اطلاعات به کل سیستم استفاده میشه و در صورتی که توی یه پروژه میخواین از یه ایمیل و Username دیگه استفاده کنید میتونین از این دستور بدون آپشن --global استفاده کنید

راه اندازی پروژه جدید:

برای شروع یه پروژه جدید، وارد دایرکتوری که میخواین پروژه تون اونجا قرار بگیره میشین (یعنی مسیر ترمینال باید اون دایرکتوری باشه) و بعد دستور زیر رو وارد میکنید.

git init
Initialized empty Git repository in /Users/user/myproject/.git/

بهتون تبریک میگم شما اولین پروژه گیت خودتون رو استارت زدین! اگه خروجی ترمینال تون توی قسمت اخر که آدرس پروژه رو نشون میده، دقیقا با آدرس پروژه ما یکی نیست نگران نباشین، دایرکتوری شما روی مسیر دیگه ای از سیستم تون قرار داره.

  • گیت از الان به بعد کد های شما رو بررسی میکنه و با هر تغییر میتونید از گیت استفاده کنید
  • گیت یک فولدر مخفی به نام .git توی پوشه ای که گیت رو استارت زدین میسازه که تغییرات اونجا ثبت میشن و اطلاعات مهمی مثل ایمیل شما و... در اونجا ذخیره میشه

حالا به فولدر مون یه فایل جدید به اسم index.html اضافه میکنیم.

git status

از این دستور برای مشاهده وضعیت دایرکتوری گیت استفاده میکنیم.

git status
On branch master

No commits yet

Untracked files: (use "git add ..." to include in what will be committed) index.html

nothing added to commit but untracked files present (use "git add" to track)
  • فایل ها برای گیت به دو دسته تقسیم میشن:
  • 1. فایل های Tracked : این فایل ها فایل هایی هستند که از قبل به پروژه اضافه شدند و تغییراتی روشون انجام شده و در مخزن گیت وجود دارند و به آن اضافه شده اند.
  • 2. فایل های Untracked : فایل هایی هستند که در دایرکتوری ما وجود دارند اما به مخزن گیت هنوز اضافه نشده اند. فایل هایی که برای اولین بار به دایرکتوری اضافه می شوند در این وضعیت قرار میگیرن!

git Staging Environment

به محیطی که فایل های Tracked و Untracked در اون قرار میگیرن تا برای مرحله ذخیره سازی در مخزن بررسی بشن، محیط Stage گفته میشه. توی این محیط فایل ها یه بررسی میشن تا اگه نیاز به تغییراتی هست که قبل از کامیت و ذخیره شدن توی مخزن به عنوان یه مرحله، باید انجام بشه، اعمال بشه و بعد از اون هم کامیت شدن اتفاق میوفته. برای اینکه فایل هامون رو استیج کنیم از دستور زیر استفاده میکنیم.

git add

از دستور git add برای استیج کردن فایل ها استفاده میشه.

git add filename

مثال

git add index.html

حالا که تغییرات فایل رو به محیط استیج بردیم، وضعیتش رو بررسی میکنیم.

git status
On branch master
No commits yet
Changes to be committed:
    (use "git rm --cached ..." to unstage)
    new file: index.html

اضافه کردن همه تغییرات به محیط استیج:

بیاین به دایرکتوری پروژه یک فایل برای معرفی پروژه به اسم README.md و یک فایل برای استایل صفحه به اسم style.css داخل یک فولدر به نام css اضافه کنیم.

index.html:

<!DOCTYPE html>
<html>
<head>
<title>Hello World!</title>
<link rel="stylesheet" href="./css/style.css">
</head>
<body>
<h1>Hello world</h1>
<p>This is the first file in my new Git Repo</p>
</body>
</html>

css/style.css:

body {
    background-color: lightblue;
}
h1 {
    color: navy;
    margin-left: 20px;
}

README.md:

# hello-world
Hello World repository for Git tutorial
This repository is built step by step in the tutorial.

حالا برای اضافه کردن همه تغییرات به محیط استیج ار یکی از سه دستور زیر استفاده میکنیم. هر سه دستور دقیقا همه ی تغییرات ما رو به محیط استیج اضافه خواهند کرد.

git add -A
git add --all
git add .

حالا وضعیت مخزن رو بررسی میکنیم:

git status
On branch master

No commits yet

Changes to be committed:
  (use "git rm --cached ..." to unstage)
      new folder: css/
      new file: README.md
      new file: index.html

هر سه فایل به محیط استیج اضافه شدن و آماده این هستن که ما اونها رو commit کنیم.

git commit

از اونجایی که کار ما تموم شده میخوایم که تغییرات رو از مرحله Stage به commit تبدیل کنیم.

  • commit کردن به ما این امکان رو میده تا:
  • - بتونیم تغییرات فایل ها رو ذخیره کنیم
  • - بتونیم بعدا از جایی که تغییرات رو ذخیره کردیم برگردیم و روال پروژه رو جلو ببریم
  • - باید برای هر commit یه پیام در نظر بگیریم تا هم خودمون هم افرادی که بعدا قرار هست تا از پروژه ما استفاده کنن به راحتی بتونن بفهمن که توی هر مرحله چه روال هایی انجام شده و راحت تر پروژه شون رو درک کنن و توسعه بدن!
  • - با گذاشتن یه پیغام مناسب برای هر commit افراد راحت تر روال ها رو میفهمن!
git commit -m "message"
git commit -m "First commit for our repo"
[master (root-commit) 221ec6e] First commit for our repo
3 files changed, 26 insertions(+)
create mode 100644 css/
create mode 100644 index.html
create mode 100644 README.md

نکته:

تگ -m برای دستور commit قابلیت این رو میده که بتونیم پیغامی رو برای هر کامیت ذخیره کنیم.

کامیت کردن بدون وضعیت استیج:

حالا میخوایم یک کامیت بدون بردن فایل به وضعیت استیج داشته باشیم. برای همین یه خط جدید به فایل index.html اضافه میکنیم.

index.html:

<!DOCTYPE html>
<html>
<head>
<title>Hello World!</title>
<link rel="stylesheet" href="./css/style.css">
</head>
<body>
<h1>Hello world</h1>
<p>This is the first file in my new Git Repo</p>
<span>New line for short commit</span>
</body>
</html>

حالا با برای دیدن وضعیت مخزن به صورت خلاصه از تگ --short یا -s که خلاصه ی تگ اول هست رو استفاده میکنیم.

git   status   -s
  M index.html
  • تگ هایی که برای نمایش وضعیت وقتی از تگ -s استفاده میکنیم معنای زیر را دارند:
  • ?? این علامت برای فایل هایی به کار می رود که جدید هستند یا Untracked هستند.
  • A این علامت زمانی به کار میرود که فایل ها به وضعیت stage اضافه شده اند.
  • M این علامت برای زمانی هست که تغییراتی روی فایل ها اعمال شده و فایل ها Modified شده اند.
  • D این علامت برای فایل هایی به کار میروند که حذف شده اند.

حالا با استفاده از فلگ -a بدون بردن به وضعیت استیج فایل ها رو کامیت میکنیم.

git   commit -a -m "Update index.html without staging"
[master 09f4acd] Update index.html without staging
1 file changed, 1 insertion(+)

نکته:

برای اینکه بدون بردن به وضعیت استیج فایل هارو کامیت کنیم، فایل ها باید حتما Tracked باشند و فایل های Untracked رو نمیتونیم بدون استیج کردن، ذخیره کنیم!
به طور کلی این روش برای زمانی استفاده میشه که تغییرات روی فایل ها خیلی جزئی بوده و میخوایم سریع کامیت بشن.

git commit logs

برای مشاهده تاریخچه commit ها، از دستور زیر استفاده میکنیم.

git log

Author: yourGitUser
Date:    Fri Mar 26 09:35:54 2021 +0100
    Update index.html without staging
commit 221ec6e10aeedbfd02b85264087cd9adc18e4b26

اگه تعداد لاگ های گیت تون خیلی زیاد باشه میتونین با کلید های بالا و پایین ↑, ↓ لاگ ها رو بررسی کنید و برای رفتن به آخر فایل از shift + G و برای خارج شدن از اون از کلید q استفاده کنین.

git help

اگه دستورات و طرز کار اونها یادتون رفت یا میخواستین که درباره کاربرد یه دستور و فلگ هاش به طور خلاصه مطالعه کنین، گیت برای ما دستور زیر رو تعریف کرده تا بتونیم به راحتی به طور خیلی خلاصه درباره دستورات، فلگ ها یا آپشن های هر دستور، اطلاعات بگیریم.

git command --help

با دستور بالا و قرار دادن نام دستور مورد نظرمون به جای کلمه command میتونیم تمامی آپشن هایی که برای اون دستور در نظر گرفته شده رو مشاهده کنیم.

git help -a
git help --all

با زدن دستور بالا، لیست تمامی دستوراتی که برای گیت وجود دارن رو میتونیم مشاهده کنیم.

git branch

branch ها در گیت، یک ورژن یا نسخه جدید از پروژه اصلی هستن. بیاین با یه مثال این مورد رو بررسی کنیم:
فرض کنید که شما یک پروژه بزرگ دارین و میخواین که برای این پروژه، تغییرات جدید اعمال کنین:

  • بدون استفاده از گیت:
  • - یک کپی از کل پروژه میگیریم تا روی پروژه اصلی تغییرات ندیم که به مشکل بخوریم.
  • - شروع به کار کردن روی دیزان پروژه میکنیم و قسمتی که میخوایم تغییر بدیم رو پیدا میکنیم و روی اون تغییرات رو شروع میکنیم.
  • - از تمامی وابستگی هایی که نسبت به قطعه کدی که ما تغییرش دادیم مطمئن میشیم تا به باگ نخوریم.
  • - یهو به یه مشکل جدی توی کل پروژه برخورد میکنیم چون تغییراتی که دادیم باعث شده یه جایی از پروژه باگ ایجاد بشه. بخاطر همین اول سراغ باگ میریم تا اون رو حل کنیم.
  • - بعد از رفع مشکل تمامی تغییرات رو ذخیره میکنیم و برای کد ها کامنت مینویسیم که تغییر داده شدن.
  • - روی دیزان که هدف اصلی مون بود شروع به کار میکنیم و کار اینجا رو هم تموم میکنیم
  • - یک کپی از فایل جدید میگیریم تا برای پابلیش کردن به جای فایل قبلی روی سرور ببریم.
  • - فایل جدید رو با فایل قبلی عوض میکنیم و خوشحال و راضی از همه چی به استراحت میریم اما...
  • - بعدا مشخص میشه که یه باگ توی پروژه به خاطر تغییرات مون پیش اومده و باید تمام این مراحل رو با پیچیدگی بیشتر از اول انجام بدیم.
  • استفاده از گیت:
  • - یک branch جدید ایجاد میکنیم و اسمش رو برای مثال new-design میزاریم. که کاملا از پروژه اصلی جداست و به اون آسیبی نمیزنه
  • - حالا به یه باگ اساسی توی پروژه خوردیم که باید هر چه سریعتر رفع بشه.
  • - یک branch جدید از پروژه اصلی به نام small-error-fix میسازیم و باگ رو اونجا رفع میکنیم
  • - برگردیم به branch که برای تغییر دیزاین پروژه داشتیم و کار خودمون رو اینجا تموم کنیم.
  • - حالا این branch رو با نسخه اصلی پروژه ادغام یا merge میکنیم. ممکنه برای اینکه باگ قبلا با پروژه اصلی همراه شده و اون رو رفع کردن به ما یه هشدار بده ولی ما میتونیم خیلی راحت ادغام رو با انجام تغییرات کوچیک انجام بدیم.

branch ها به ما این امکان رو میدن تا بدون دستکاری پروژه اصلی، روی قسمتی از پروژه به صورت مجزا کار کنیم و بعدا اون رو به پروژه اصلی اضافه کنیم.

شما میتونید روی شاخه های مختلف، branch های مختلف، به صورت جدا کار کنین بدون اینکه روی هم اثر بزارن.

branch ها خیلی سبک تر از کپی یک نسخه جدید از کل پروژه هستن!

git new branch

بیاین فایل index.html رو تغییر بدیم. اما نمیخوایم که روی پروژه اصلی این کار انجا بشه. برای این کار از branch استفاده میکنیم. برای همین کار بیاین یک branch جدید ایجاد کنیم:

git branch change-index-page

برای اینکه مطمئن بشیم branch جدید ساخته شده یا نه، از دستور زیر کمک میگیریم. این دستور لیست تمامی branch هایی که داریم رو به ما نشون میده.

git branch
  change-index-page
  * master

کنار نام یک branch به نام master یک علامت * میبینید. این علامت به معنی اینه که تغییراتی که انجام بدین روی این branch اتفاق میوفته و نشون میده شما در حال حاضر روی کدوم branch هستین!

حالا که شاخه جدید پروژه مون رو ساختیم، تغییرات رو روی اون اعمال کنیم. برای این کار اول باید branch پروژه مون رو عوض کنیم:

git checkout <branch name>

حالا که شاخه جدید پروژه مون رو ساختیم، تغییرات رو روی اون اعمال کنیم. برای این کار اول باید branch پروژه مون رو عوض کنیم:

دستور checkout دستوریه که برای ما تعویض branch ها روانجام میده. بیاین با استفاده از این دستور به branch جدیدی که ساختیم تعویض شاخه کنیم.

git checkout change-index-page
Switched to branch 'change-index-page'

تعویض محیط کار یا work space ما انجام شد و تغییرات جدید روی این محیط فقط اعمال میشن.
حالا بیاین فایل index.html رو تغییر بدیم:

index.html:

<!DOCTYPE html>
<html>
<head>
<title>Hello World!</title>
<link rel="stylesheet" href="./css/style.css">
</head>
<body>

<p>This is the first file in my new Git Repo</p>
<span>New line for short commit</span>
<p>This is a new line from new branch called 'change-index-page' </p>

شما تغییرتون رو روی فایل index.html انجام دادین، حالا بیاین وضعیت رو بررسی کنیم:

حالا بیاین تغییرات رو ذخیره کنیم:

git add index.html
git commit -m "add new line in index.html in new branch"
[change-index-page 0312c55] add new line in index.html in new branch
2 files changed, 1 insertion(+)

نکته :

برای اینکه یک branch جدید بسازیم و همزمان به اون سویچ کنیم و محیط کارمون به اون منتقل شه میتونیم از آپشن -b برای دستور checkout استفاده کنیم و نام branch جدیدمون رو که میخوایم ساخته بشه بنویسیم.
این آپشن برای ما branch جدید رو میسازه و به محیط کار مارو به اون منتقل میکنه.

تعویض محیط کار:

همونطوری که بالاتر محیط کارمون رو از master به change-index-page تغییر دادیم، الان هم از این branch به شاخه اصلی پروژه یا master سویچ میکنیم.

git checkout master
Switched to branch 'master'

حالا اگه فایل index.html رو بررسی کنین، میبینید که محتوای اون با قبل فرقی نکرده و هنوز هم همون محتوا رو داره.

index.html:

حالا فرض کنید به باگ عجیبی توی فایل index.html رسیدیم که باید سریعا رفع شه. حواستون باشه که ما الان روی master هستیم و میخوایم این ارور رو رفع کنیم.

git checkout -b fix-error
Switched to a new branch 'fix-error'

حالا که روی یک branch جدا از master هستیم، اررور رو رفع میکنیم و تغییرات رو اضافه میکنیم.

تغییرات رو ثبت میکنیم:

git commit -a -m "updated index.html with fixing the error"
[fix-error dfa79db] updated index.html with fixing the error
1 file changed, 1 insertion(+), 1 deletion(-)

برای اینکه تغییرات ثبت شده رو به master منتقل کنیم، باید branch ها رو با هم یکی یا merge کنیم. برای این کار اول به master میریم و از اونجا برای merge اقدام میکنیم.

git checkout master
Switched to branch 'master'

git merge <branch name>

برای یکی کردن دو محیط کار، اول به محیط کاری که محیط کار اصلی ما هست checkout میکنیم و بعد از اونجا برای یکی کردن دو محیط کار یا branch اقدام میکنیم.

git merge fix-error
Updating 09f4acd..dfa79db
Fast-forward
  index.html | 2 +-
  1 file changed, 1 insertion(+), 1 deletion(-)

برای اینکه fix-error از شاخه ی master گرفته شده بود، و هیچ تغییر دیگه ای روی شاخه master انجام نشده بود، عملیات یکی کردن شاخه ها رو به صورت Fast-forward انجام میده چون شاخه ی fix-error رو به عنوان ادامه دهنده ی master میشناسه.

git branch -d <branch name>

برای اینکه ما تغییرات fix-error رو به شاخه اصلی پروژه اضافه کردیم، به اون شاخه نیازی نداریم و میتونیم اون رو حذف کنیم تا مخزن پروژه تمیز تر بشه.

git branch -d fix-error
Deleted branch emergency-fix (was dfa79db).

Merge conflict

حالا میخوایم دو شاخه ی master و change-index-page رو با هم یکی کنیم. اگه یادتون باشه ما توی change-index-page هم تغییرات داده بودیم و یک خظ به اون فایل اضافه کرده بودیم. ولی از اون موقع تاحالا ما روی index.html که روی شاخه اصلی پروژه بود هم تغییرات دادیم. حالا اگه بخواهیم این دو شاخه رو یکی کنیم، به پیچیدگی برخورد میکنیم. و گیت نمیدونه که کدوم تغییرات رو اولویت قرار بده و نمیتونه تغییرات رو یکی کنه برای همین از ما کمک میخواد!

git merge change-index-page
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.

نکته:

گیت تا جایی که بتونه عملیات یکی کردن رو انجام میده، یعنی اگه توی یه فایل دیگه مثل مثلا css/style.css تغییر داشتیم، چون توی branch/master مون تغییرش نداده بودیم، اونها رو با هم یکی میکرد ولی فایل هایی که نتونه یکی کنه رو بهمون میگه خودمون انجام بدیم.

حالا اگه وضعیت رو بررسی کنیم، به ما میگه که روی یک فایل دو تغییر انجام شده:

GitHub

گیتهاب جاییه که افراد، پروژه هاشون رو اون تو برای نمایش به افراد دیگه یا شرکت ها برای اینکه بتونن روی یک پروژه با برنامه نویس هاشون به صورت تیمی کار کنن روی گیتهاب یا ممکنه یک سرور گیت دیگه ای، پروژه رو تعریف کنن و برای هر فرد یک سطح دسترسی قرار بدن تا اون فرد بتونه روی قسمت خودش توسعه بده و بعدا از مدیر سیستم بخواد تا کد اون رو با پروژه اصلی یکی کنه.

برای ساختن یک repository روی گیتهاب باید اول یک اکانت گیتهاب بسازین و با اون وارد گیتهاب بشین، بعد روی قسمت add new repository کلیک کنید.

حالا فیلد های مربوط به اسم مخزن و توضیحات رو پر کنین رو پایینتر روی دکمه Create Repository کلیک کنید.

بقیه فیلد هایی که میبینید رو بعدا بررسی میکنیم و فعلا باهاشون کاری نداشته باشین!

حالا که repository جدیدمون ساخته شد، پروژه مون که از قبل تعریف کردیم روی گیتهاب آپلود میکنیم. برای این کار URL داده شده رو کپی کنین!

git push

این دستور به ما اجازه میده تا تغییرات ها مون رو روی یک ریموت آپلود کنیم. (پروژه مون رو روی یک ریموت قرار بدیم).

حالا مسیر کپی شده رو توی ترمینال گیت مون که داخل مسیر پروژه هست به پروژه به عنوان یک remote تعریف میکنیم

git remote

معنی remote به معنای دورکاری توی لغت نامه هست اما داخل گیت میشه پروژه ای که روی یک سرور یا جایی خارج از محیط گیت ما قرار داره، این محیط خارجی حتی میتونه روی کامپیوتر خود ما باشه!
برای همین ما باید این محیط خارجی رو به گیت پروژه مون تعریف کنیم تا بتونه با اون ارتباط بگیره، کد ها رو از اون دریافت یا روی اون آپلود کنه و...

  • نکات مهم remote
  • - ما میتونیم remote های مختلفی رو به پروژه اصافه کنیم ولی هر کدوم از اینها باید یه اسم داشته باشن!
    به صورت پیش فرض و قراردادی، اون remote که به عنوان ریموت اصلی ما هست اسمش رو origin قرار میدن.
  • برای اضافه کردن remote جدید به پروژه باید از کلمه add استفاده کنیم.
  • برای اضافه کردن remote جدید باید یک مسیر برای اون ریموت در اختیار ما باشه که بتونیم اون رو اضافه کنیم.
git remote add origin https://github.com/Amirmahdikahdouii/git-tutorial-farsi.git

-u option

این آپشن تنها یک بار برای هر آپلود به کار میره و تعیین میکنه که بعدا اگه دستور git push خالی زدم، تغییرات رو روی ریموت مورد نظر و از branch مورد نظر اعمال کن.

میبینید که فایل هامون با موفقیت آپلود شدن و از گیتهاب میتونین اونها رو بررسی کنین.
